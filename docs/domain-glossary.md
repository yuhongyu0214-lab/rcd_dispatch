## 重要说明：司机状态双套命名对照

PRD（第 9.2 节）与 dispatch-rules-v1.md（第二节）对司机状态使用了两套命名，本文件统一如下：

| 代码枚举（唯一依据） | PRD 中文名 | dispatch-rules 简码 | 含义 |
|---------------------|-----------|---------------------|------|
| `OFFLINE` | 离线 | — | 司机不在线，不参与调度 |
| `S1` | 在线空闲（门店） | S1：门店空闲 | 位于门店，当前无接单 |
| `S2` | 在线空闲（返程） | S2：返程空闲 | 正在返回门店途中，无接单 |
| `S3` | 门店忙碌 | S3：门店忙碌 | 在门店但正在处理任务 |
| `S4` | 执行中 | S4：订单忙碌 | 正在执行订单 |
| `UNAVAILABLE` | 不可用 | — | 临时不可接单（请假、故障等） |

> **代码实现以枚举值为准**，不使用 PRD 中文名或 S1–S4 简码作为数据库存储值。

## 术语定义

---

### 订单（Order）

**定义**：系统中的调度工单，代表一次需要司机执行的人车交接任务。

**来源**：
- 手动导入（Excel / CSV）
- 哈啰 API / 三方 SaaS API 自动同步
- （V2）官方平台 API 直连

**核心字段**：订单 ID、订单类型、归属门店、车型、车牌号、取车点、还车点、预约时间、状态、导入批次号。

**订单类型**（`OrderType` 枚举）：

| 枚举值 | 中文名 | 说明 |
|--------|--------|------|
| `STORE_PICKUP` | 到店取车 | 客户到门店取车 |
| `STORE_RETURN` | 到店还车 | 客户到门店还车 |
| `DOOR_DELIVERY` | 送车上门 | 司机将车送到客户指定位置 |
| `DOOR_PICKUP` | 上门取车 | 司机到客户位置取回车辆 |

> 判断是否为上门类：`type === 'DOOR_DELIVERY' || type === 'DOOR_PICKUP'`

**参考**：`./order-lifecycle.md`、`./import-template.md`

---

### 司机（Driver）

**定义**：系统中负责执行订单的人员角色，拥有移动端账号，可接受派单并上报位置。

**司机状态**（`DriverStatus` 枚举）：

| 枚举值 | 含义 | 参与调度 |
|--------|------|---------|
| `OFFLINE` | 离线，不在线 | 否 |
| `S1` | 门店空闲，位于门店无接单 | 是（优先级最高） |
| `S2` | 返程空闲，返回途中无接单 | 是 |
| `S3` | 门店忙碌，在门店处理中 | 是 |
| `S4` | 订单忙碌，正在执行订单 | 是（优先级最低） |
| `UNAVAILABLE` | 不可用，临时不可接单 | 否 |

**位置来源**：司机移动端每 3–5 分钟上报一次经纬度，写入 `driver_locations` 表，调度引擎取最新一条。

---

### 车辆（Vehicle）

**定义**：系统中参与调度的车辆资产，与订单通过车牌号关联。

**车辆状态**（`VehicleStatus` 枚举）：

| 枚举值 | 中文名 | 含义 |
|--------|--------|------|
| `AVAILABLE` | 可用 | 可被分配给新订单 |
| `PRE_ASSIGNED` | 预占用 | 已通过车牌预排关联某订单，未正式执行 |
| `IN_USE` | 执行中 | 正在被某订单使用 |
| `UNAVAILABLE` | 不可用 | 临时不可分配（维修、故障等） |

**说明**：V1 版本车辆通过车牌号预排，允许调度员人工改排。车辆实时 GPS 位置在 V1 中预留字段（`gpsLat`、`gpsLng`），不强依赖实时数据。

---

### 门店（Store）

**定义**：业务运营的物理场所，是订单和司机归属的基本组织单元。

**作用**：
- 订单必须归属某个门店（`storeId`）
- 调度引擎只在同一门店维度内筛选候选司机
- 负载均衡策略以门店为维度计算（参见 dispatch-rules-v1.md 4.1(3)）

---

### 派单（Assign / Dispatch）

**定义**：调度员将某条 `PENDING` 订单分配给特定司机的操作，分为两种方式：

| 方式 | 说明 |
|------|------|
| 推荐派单 | 系统通过调度引擎计算 Top N 候选，调度员从推荐列表中确认 |
| 手动派单 | 调度员直接指定司机，不经过推荐引擎 |

**触发结果**：订单状态 `PENDING` → `ASSIGNED`，写入 `assignments` 表，触发调度锁。

---

### 改派（Reassign）

**定义**：对已处于 `ASSIGNED` 或 `ACCEPTED` 状态的订单，将执行司机更换为另一位司机的操作。

**规则**：
- 改派必须填写原因，写入 `operation_logs`
- 历史 assignment 记录保留，新建一条新的 assignment 记录
- 原司机状态恢复为改派前状态，调度锁释放
- 新司机立即进入 `ASSIGNED` 状态

---

### 撤回（Withdraw）

**定义**：调度员主动将 `ASSIGNED` 状态的订单回收，终止当前派单，订单重新进入 `PENDING` 状态。

**与改派的区别**：撤回不指定新司机，订单回到待分配池等待重新派单；改派同时指定新司机。

**触发结果**：订单状态 `ASSIGNED` → `RECYCLED` → `PENDING`，调度锁释放，写操作日志。

---

### 回收（Recycle）

**定义**：订单因异常（司机拒单、超时未接单、人工干预）从非终态回到 `PENDING` 的机制。

**触发场景**：
- 司机拒单（`ASSIGNED` → `RECYCLED`）
- 超时未接单（`ASSIGNED` 状态超过阈值，自动触发）
- 调度员撤回（`ASSIGNED` → `RECYCLED`）
- 执行中异常（`IN_PROGRESS` → `RECYCLED`，人工操作）

**说明**：`RECYCLED` 是中间过渡状态，正常秒级内自动变为 `PENDING`。

---

### ETA（Estimated Time of Arrival）

**定义**：预计到达时间，指候选司机从当前位置驾车到达订单取车点所需的分钟数。

**计算方式**：调用高德驾车路径规划 API（`/v3/direction/driving`），返回 `duration` 字段（秒），换算为分钟。

**用途**：
- 作为调度推荐排序的核心依据（参见 dispatch-rules-v1.md 第四节）
- ETA ≥ 120 分钟时，调度结果标记为 `MANUAL`（人工处理）

**注意**：高德 API 使用服务端 Key（`AMAP_SERVER_KEY`），不同于地图看板使用的 JS Key。

---

### 调度锁（Dispatch Lock）

**定义**：派单确认瞬间对司机施加的并发保护机制，防止同一司机被同时派给多条订单。

**实现方式**：在 `/api/dispatch/confirm` 接口中使用 Prisma 数据库事务，原子性写入 `assignments` 记录并更新司机状态。

**生命周期**：
- 加锁：调度员确认派单时
- 释放：订单完成 / 撤回 / 回收 / 司机拒单时

---

### 导入批次号（Import Batch ID）

**定义**：每次导入操作（无论手动还是 API）系统自动生成的唯一标识符，用于关联同批次的所有订单记录和日志。

**格式**：UUID v4，系统自动生成，不需要用户填写。

**用途**：
- 导入结果页展示当次成功 / 失败统计
- `operation_logs` 中关联导入事件
- 排查导入问题时的追踪依据

---

### 推荐理由（Recommendation Reason）

**定义**：调度引擎为每位候选司机生成的中文说明，展示给调度员作为派单决策依据。

**格式示例**：
- `门店空闲，预计到达 8 分钟`
- `返程空闲，预计到达 15 分钟`
- `门店忙碌，预计到达 12 分钟，负载惩罚 +7 分钟（已接 1 单）`

**生成位置**：`lib/dispatch/sort.ts` 中的 `buildReason()` 函数。

---

### 负载惩罚（Load Penalty）

**定义**：到店类订单调度中，对已接多单司机施加的时间惩罚值，用于实现负载均衡。

**公式**：`负载惩罚 = 当前司机已接到店类订单数 × 惩罚系数`

**默认系数**：7 分钟（范围建议 5–10，通过环境变量 `STORE_ORDER_PENALTY` 配置）。

**适用范围**：仅到店类订单（`STORE_PICKUP` / `STORE_RETURN`），上门类订单不加负载惩罚。

---

### 待分配池（Pending Pool）

**定义**：所有处于 `PENDING` 状态的订单集合，是调度员日常工作的起点。

**来源**：新导入的订单、被撤回的订单、被回收的订单。

**展示位置**：调度员后台订单列表页，可按状态筛选。

---

### trace_id（追踪 ID）

**定义**：每个 API 请求生成的唯一标识符，贯穿该请求的全部日志记录，用于跨接口排查问题。

**格式**：UUID v4，每次请求入口自动生成。

**传递方式**：写入响应头 `X-Trace-Id`，同时写入所有相关日志条目。
