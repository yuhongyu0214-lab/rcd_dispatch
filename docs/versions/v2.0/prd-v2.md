# 人车单智能调度系统 PRD V2

> 产品版本：`RCD-PRD-V2.0-20260713`
> 产品负责人：喻红宇
> 状态：产品口径冻结；进入数据模型与接口设计阶段
> 适用范围：调度员 Web、司机执行端接口、调度引擎、订单接入与数据追溯

## 1. 一句话定义

让调度员和司机在一张全局地图上看见所有上班司机的位置与订单位置，并用每名司机的 A/B/C 三个工单时间轴持续自动重排，减少司机在订单之间的空驶与等待时间。

## 2. 问题、目标与边界

### 2.1 当前问题

- 订单、人员位置和工作占用分散，无法判断每一分钟被什么工作占用。
- V1 没有把高德 ETA、实时人员位置和工单时间轴组成一个可操作闭环。
- 单次 Top N 推荐不能处理新增订单对所有司机后续动线的影响。
- 外部订单来源与内部调度状态耦合，更换数据源时迁移成本高。

### 2.2 V2 目标

1. 降低司机在前后工单之间的空驶时间。
2. 保证司机当前任务和未来两个任务紧密、可执行。
3. 对预计迟到超过 30 分钟的订单提前预警并推动人工修正。
4. 让自动排程、人工改派和订单修改全部可追溯。
5. 未来切换外部公司 API 时，不重写调度引擎、地图、ETA、Redis 和前端。

### 2.3 暂不纳入

- 不计算司机全天全局最优路线，只计算 A/B/C。
- 车辆不参与 V2 派单匹配和评分。
- 暂不制作外部公司端或第一版司机端视觉演示。
- 不建设计费、结算、车辆运营优化和复杂多组织权限。

## 3. 马斯克五步法结论

| 步骤 | V2 决策 |
|---|---|
| 质疑 | 删除“必须先 Top N 再人工确认”“车辆必须匹配”“司机必须接单”等 V1 假设 |
| 精简 | 每名司机只计算 A/B/C；五个模块只用固定时长和组合总时长 |
| 优化 | 用高德 ETA、手机实时定位、承诺取车时间和 30 分钟上限优化空驶 |
| 快速行动 | 新订单和业务事件立即局部重排，另每 10 分钟校验一次 |
| 自动化 | 默认自动分配；调度员保留改派、修改时间地点和解除锁定能力 |

## 4. 核心业务模型

### 4.1 工单

同一名司机完整执行一个工单：

```text
到达取车点 → 执行所选固定模块 → 驾驶车辆 → 到达送达点 → 完成
```

工单包含承诺取车时间、取车地点、送达地点、订单快照、服务模块和执行记录。完成后，下一个 ETA 起点以司机手机实时位置为准，不假设司机停留在固定门店。

### 4.2 A/B/C 队列

| 槽位 | 含义 | 自动调整规则 |
|---|---|---|
| A | 当前正在执行或即将执行的工单 | 未出发可调；出发后自动冻结 |
| B | 下一工单 | 可滚动重排，手动锁定除外 |
| C | 下下工单 | 可滚动重排，手动锁定除外 |

不存在 C 之后的“自由规划区”。其他订单留在未分配池，等待后续进入某名司机的 B 或 C。

### 4.3 固定服务模块

| 模块 | 默认时长 |
|---|---:|
| 充电 | 30 分钟 |
| 加油 | 5 分钟 |
| 洗车 | 10 分钟 |
| 交车手续 | 10 分钟 |
| 还车手续 | 5 分钟 |

- 司机领取工单后可自由多选，模块无顺序约束。
- 模块时长累计进入工单时间轴。
- 工单执行后仍允许调整模块；调整立即重算后续 ETA 并写日志。
- 不记录模块级实际耗时，只记录工单级到达和完成时间。
- 可用时间小于模块总时长时显示红色和“时间不足”；否则显示绿色和“时间充足”。

## 5. 时间和 ETA 定义

```text
计划占用时间 = 固定模块总时长 + 高德“取车点 → 送达点”驾车 ETA
实际占用时间 = 点击到达时间 → 点击完成时间
衔接 ETA = 前一工单结束位置（或手机实时位置）→ 下一工单取车地点
```

- 点击“到达”即表示工单可以立即开始，实际计时从此刻开始。
- 调度目标是在时间约束内，最小化所有司机 A→B、B→C 的衔接 ETA。
- 以订单承诺取车时间作为约束，预计迟到不得超过 30 分钟。
- 若任何安排都会超过 30 分钟，订单进入“确认无法准时”预警。

### 5.1 可行性判定（冻结）

```text
slackMinutes = promisedPickupAt - projectedPickupAt
（安全余量 = 承诺取车时间 - 预计到达取车点时间，单位分钟）
```

| 可行性 | 判定 |
|---|---|
| `UNKNOWN` | 尚未计算，或缺有效位置/ETA |
| `NORMAL` | `slackMinutes >= 10` |
| `AT_RISK` | `-30 <= slackMinutes < 10` |
| `INFEASIBLE` | `slackMinutes < -30` |

- 采用 10 分钟安全余量，避免 0–10 分钟出现定义空档。
- 可行性只有以上四个值；预警的处理状态（`OPEN / RESOLVED`）属于 `DispatchAlert`，不属于可行性维度。

## 6. 自动滚动调度

### 6.1 候选范围（冻结）

```text
候选司机 = onShift = true
        && availability = AVAILABLE
        && locationFreshness = FRESH
```

- 三条件缺一不可：当班（班次）、可用（`DriverAvailability`，调度员设置的人为停派开关）、定位有效（位置新鲜度）。
- 车辆信息可展示，但不进入过滤和评分。
- 已有 A/B/C 的锁定状态和执行状态必须被尊重。

### 6.2 排序原则

1. 先处理承诺取车时间更早的订单。
2. 排除预计迟到超过 30 分钟的不可行组合。
3. 在可行组合中，优先选择衔接 ETA 最短的司机与槽位。
4. 只重排 A/B/C，不承诺全天全局最优。

### 6.3 触发机制

以下事件立即触发受影响范围重排：

- 新订单进入、取消或订单时间/地点变化。
- 司机上班、下班、位置恢复或位置失效。
- 司机点击出发、到达、完成。
- 服务模块组合变化。
- 调度员派单、改派、撤回或解除锁定。

此外：

- 每 10 分钟执行一次基线校验。
- 高德 ETA 相比上次有效值变化超过 10 分钟时触发重排。
- 10 分钟周期不是定位频率；司机位置由执行端持续更新。

## 7. 冻结、改派和权限

### 7.1 自动冻结

- 司机点击“出发”时进入导航，A 自动冻结为 `AUTO_FROZEN`。
- 算法不得移动已出发工单。
- 调度员在司机到达前仍可手动改派，必须记录原因和前后值。
- 点击“到达”后任何人都不得改派。

### 7.2 手动分配

- 系统默认自动分配。
- 调度员有手动分配和改派权限。
- 手动分配后进入 `MANUAL_LOCKED`，算法不得覆盖，直到解除或司机出发。
- 司机下班时，所有尚未出发的自动或手动锁定工单释放并重新调度。
- 调度员可将司机设为不可用（`availability = UNAVAILABLE`）：未出发工单全部释放并重排；执行中（已出发/已到达）工单继续执行到完成；恢复可用需调度员再次设置。

### 7.3 司机权限

- 司机无权拒绝、改派订单。
- 司机只能对自己的工单选择模块、出发、到达和完成。
- 司机处于已出发或执行中时不得下班。
- 无执行中任务时允许直接下班，未来工单自动释放。

## 8. 不可行订单处理闭环

1. 建立或更新 `INFEASIBLE` 预警（`DispatchAlert`，状态 `OPEN`）。
2. 地图、订单池、司机时间轴和预警中心显示特殊标注。
3. 调度员可直接修改取车时间、取车地点或送达地点。
4. 修改无需二次确认，保存后立即生效并重算 ETA。
5. 日志记录修改人、时间、原值、新值、原因和 traceId。
6. 新方案可行后，预警状态转为 `RESOLVED`，历史记录保留；订单可行性按 5.1 重新计算为 `NORMAL`、`AT_RISK` 或 `UNKNOWN`。

预警是需要处理的业务对象，不得只写一条普通日志。

## 9. 可视范围

### 9.1 调度员 Web

- 查看所有上班司机的实时位置和全部订单位置。
- 查看所有司机的 A/B/C、未分配订单和每段时间占用。
- 查看高德 ETA、位置更新时间、锁定状态和不可行预警。
- 手动派单、改派、解除锁定，修改订单时间或地点。

### 9.2 司机端

- 查看所有上班司机的位置；全局位置视图不能删除。
- 只查看自己的 A/B/C 详情，以及尚未分配的订单。
- 不查看其他司机负责订单的详细内容。
- 不能拒单或改派，只能由调度员处理。

## 10. 生命周期

V2 将外部状态、内部执行、可行性、锁定和槽位分开。

```text
UNASSIGNED → PLANNED → EN_ROUTE → IN_SERVICE → COMPLETED
     ↑           |
     └───────────┘  （到达前可释放或改派）

UNASSIGNED / PLANNED / EN_ROUTE → CANCELLED
```

| 状态 | 业务含义 |
|---|---|
| `UNASSIGNED` | 已入内部系统，未排入司机 A/B/C |
| `PLANNED` | 已自动或手动分配，尚未出发 |
| `EN_ROUTE` | 司机已出发，自动冻结并导航 |
| `IN_SERVICE` | 已到达，开始计时且不可改派 |
| `COMPLETED` | 已完成，结束计时 |
| `CANCELLED` | 已取消 |

正交维度：

| 维度 | 枚举 |
|---|---|
| 可行性 | `UNKNOWN / NORMAL / AT_RISK / INFEASIBLE`（判定见 5.1） |
| 锁定 | `NONE / AUTO_FROZEN / MANUAL_LOCKED` |
| 槽位 | `NONE / A / B / C`（`NONE` 表示未占任何槽位，避免与执行状态 `UNASSIGNED` 同名） |

预警处理状态 `OPEN / RESOLVED` 属于 `DispatchAlert` 实体，不是订单维度。

### 10.1 状态流转矩阵（冻结）

| 当前状态 | 操作 | 目标状态 | 允许角色 | 副作用 | 非法时响应 |
|---|---|---|---|---|---|
| `UNASSIGNED` | 自动分配 | `PLANNED` | 调度引擎 | 建立 Assignment、写入槽位与计划时间、递增 `planVersion`、日志 | — |
| `UNASSIGNED` | 手动分配 | `PLANNED` | 调度员 | 同上，且锁定 = `MANUAL_LOCKED`，记录原因 | — |
| `PLANNED` | 撤回/释放 | `UNASSIGNED` | 调度员、调度引擎（下班或设为不可用时释放） | 槽位 = `NONE`、锁定 = `NONE`、触发重排、日志 | — |
| `PLANNED` | 改派 | `PLANNED`（新司机） | 调度员 | 旧 Assignment 结链、新 Assignment 建立、记录原因与前后值、触发重排 | — |
| `PLANNED` | 出发 | `EN_ROUTE` | 司机（仅本人工单） | 锁定 = `AUTO_FROZEN`、记录 `departedAt`、开启导航、刷新 ETA | 400 |
| `EN_ROUTE` | 改派 | `PLANNED`（新司机） | 调度员 | 必须记录原因；原司机任务释放；触发重排 | — |
| `EN_ROUTE` | 到达 | `IN_SERVICE` | 司机（仅本人工单） | 记录 `arrivedAt`、实际计时开始、此后任何角色改派均被服务端拒绝 | 400 |
| `IN_SERVICE` | 改派 | — | 任何角色 | 禁止 | 400，含 `currentStatus/targetStatus` |
| `IN_SERVICE` | 完成 | `COMPLETED` | 司机（仅本人工单） | 记录 `completedAt`、实际计时结束、以手机实时位置为下一工单 ETA 起点、触发重排 | 400 |
| `UNASSIGNED / PLANNED / EN_ROUTE` | 取消 | `CANCELLED` | 来源系统（经 Adapter）、调度员 | 释放司机与槽位、递增受影响司机 `planVersion`、`OPEN` 预警转 `RESOLVED`、触发重排、日志（端点与幂等语义见 API 契约 V2 §2.1） | — |
| `IN_SERVICE / COMPLETED` | 取消 | — | 任何角色 | 禁止 | 400，含 `currentStatus/targetStatus` |
| `COMPLETED / CANCELLED` | 任何流转 | — | 任何角色 | 终态，禁止流转 | 400 |

- 矩阵未列出的任何流转组合一律非法，返回 400 并包含 `currentStatus` 和 `targetStatus`。
- 重复提交已生效的同一流转（如重复点击“到达”）按幂等处理，见 API 契约 V2。

V1 的状态暂不删除，在数据迁移阶段通过兼容层映射。

## 11. 数据来源边界

- 外部公司 API、浏览器插件或临时 RDS 只提供订单主数据。
- `OrderSourceAdapter` 负责校验、映射、幂等和错误隔离。
- `CanonicalOrder` 是内部唯一标准订单格式。
- 内部 PostgreSQL/RDS 保存订单快照、调度、执行、预警和日志。
- Redis/Tair 保存最新位置、在线状态、ETA 短缓存和短锁。
- 高德负责地图、地理编码、路径、ETA 和导航。
- 前端只调用内部 API，不依赖外部数据库字段。

详见 [数据架构 V2](data-architecture-v2.md)。

## 12. 非功能要求

- API 返回 `{ success, data, error, traceId }`，响应头包含 `X-Trace-Id`。
- 写操作使用事务；重排和派单必须防并发覆盖。
- 外部 API、高德或 Redis 故障不得产生假 ETA。
- 位置必须带采集时间和精度；过期位置明确标注。
- 订单来源事件幂等，同一来源版本不得重复创建订单。
- 人工修改和算法重排均可按订单、司机和 traceId 追溯。

## 13. 验收标准

1. Web 同屏显示所有上班司机、订单点位和选中司机 A/B/C 时间轴。
2. 新订单进入后，受影响司机的 B/C 自动重排并留下版本日志。
3. 出发后 A 自动冻结；到达前调度员可改派，到达后改派被拒绝。
4. 到达开始实际计时，完成停止计时。
5. 五个模块可组合、可在执行中修改，并影响后续时间轴。
6. 任何方案迟到超过 30 分钟时建立显著预警。
7. 修改订单时间或地点后立即重算并保存前后值。
8. 司机下班释放未出发任务，但执行中不能下班。
9. 更换外部订单 API 时，只替换 Adapter，不修改调度引擎核心输入。
10. 车辆字段可展示，但不影响 V2 派单结果。

## 14. V1 → V2 变化摘要

| V1 | V2 |
|---|---|
| Top N 推荐，人工确认 | 自动滚动分配，人工可覆盖 |
| 单订单选司机 | 每名司机 A/B/C 局部规划 |
| 司机接单 | 司机无拒单权，以出发/到达/完成执行 |
| 车辆参与匹配 | 车辆仅展示 |
| ETA≥120 分钟转人工 | 迟到超过 30 分钟即不可行预警 |
| 3–5 分钟定位、手动刷新 | 实时定位、事件重算、10 分钟校验 |
| 一个状态枚举承担多种含义 | 执行、可行性、锁定和槽位分维度 |
