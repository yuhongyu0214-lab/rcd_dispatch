1. 本文件、PRD V2、数据架构 V2。

## 3. 代码一致性规则

### 3.1 继续锁定

- Next.js 14 App Router + TypeScript。
- Tailwind CSS + shadcn/ui，不引入第二套 UI 框架。
- PostgreSQL + Prisma。
- Redis/Tair 只保存实时或短期数据，数据库保存业务事实。
- 高德服务端 Key 只在服务端；JS Key 只用于地图渲染。
- Pino 结构化日志，禁止 `console.log`。
- pnpm `10.11.0`，禁止 npm/yarn 和新增 `package-lock.json`。
- API 使用 `ok()` / `fail()`，统一返回 `{ success, data, error, traceId }`。
- 枚举 UPPER_SNAKE_CASE，函数/变量 camelCase，文件 kebab-case，组件 PascalCase。

### 3.2 V2 新增硬规则

- 外部订单字段只能存在于 Adapter 和来源事件层；调度代码只接收 CanonicalOrder 或内部 DTO。
- 外部原始状态不得直接写入内部执行状态。
- 执行状态、可行性、锁定和 A/B/C 槽位必须分开建模。
- 车辆字段不得进入 V2 候选过滤、评分和迟到判断。
- 自动重排必须使用 `planVersion` 或等价乐观锁。
- 已到达工单的改派必须在服务端拒绝，不能只禁用前端按钮。
- ETA 不可用时返回明确原因，不得使用假数据。
- 人工订单修改保存原值、新值、操作者、时间、原因和 traceId。
- V2 代码不得复用含 V1 业务语义的类型名而不写兼容说明。

## 4. 开发与分支边界

- V1 文件与主线暂不删除，现有 bug 修复仍按原 worktree 范围执行。
- V2 新功能使用 `feature/v2-*` 分支或独立 worktree，不把 V2 schema 与 V1 页面修改混进同一提交。
- 合并路径仍为 `feature/* → develop → main`。
- V2 顺序：文档 → schema → 内部 DTO/API → Adapter → 调度引擎 → 页面 → 观测与稳定性。
- 当前文档阶段只修改 Markdown；本版本不授权修改 `.ts/.tsx/.prisma`。
- Schema 变更先生成迁移 SQL 和 rollback SQL，再等待审查。
- 每阶段退出前通过测试、构建和业务验收，不以“页面能打开”替代闭环验收。

## 5. 设计系统 V2

### 5.1 产品气质

克制、专业、信息密集的调度控制台。禁止回退为普通蓝白 SaaS、营销页、大卡片堆叠、过圆组件和无意义动画。

### 5.3 V2 布局规则

- Web 保留“深色导航 + 左侧工作面板 + 右侧主工作区”。
- 全局地图是核心工作区，不得被订单表格完全替代。
- 调度主视图同屏展示地图和选中司机 A/B/C 时间轴。
- 时间轴区分空驶 ETA、固定模块、工单驾驶、空闲和迟到风险。
- 司机位置显示更新时间；过期时同时使用灰色、图标和文字。
- 不可行订单在地图、列表和时间轴至少两处可见，不能只藏在日志页。
- 颜色只作辅助，红/绿模块同时显示“时间不足/时间充足”。
- 页面级不滚动，密集内容使用模块内部滚动；宽表格使用横向滚动。
- 选中对象联动地图、列表和时间轴，不通过跳页破坏上下文。

### 5.4 V1 设计行为的替换

| V1 行为 | V2 行为 |
|---|---|
| 地图底部 Top N 推荐卡 | 司机 A/B/C 时间轴与重排结果 |
| 点击候选司机选中 | 系统自动排程，调度员可手动改派 |
| 车辆作为地图调度对象 | 车辆仍可查看，但不参与算法 |
| 司机“接单”操作 | 出发、到达、完成三步执行 |
| 普通 ETA 超限提示 | 持续的不可行预警和修改入口 |

## 6. API 与状态规则

- V2 API 契约必须单独版本化，不直接改写 `demo-v12-api-contract.md`。
- 内部路径保持稳定，通过 DTO 兼容，不把外部 API 结构透传给页面。
- 并发冲突返回 409 和当前 `planVersion`。
- 非法状态流转返回 400，并包含 `currentStatus` 和 `targetStatus`。
- 外部依赖故障返回可理解错误，详细原因写结构化日志。
- 所有时间使用带时区 ISO 8601；数据库统一存 UTC，前端按 Asia/Shanghai 展示。

## 7. 审查清单

### P0 阻断

- 是否跨越 worktree 或阶段范围。
- 是否绕过 CanonicalOrder 直接使用外部字段。
- 是否允许算法覆盖锁定或执行中工单。
- 是否允许到达后改派。
- 是否存在重复排程、重复派单或非幂等入库。
- 是否使用假 ETA 或把过期位置当实时位置。

### P1 严重

- API 是否统一响应并带 traceId。
- 自动/人工变更是否有结构化日志与前后值。
- 状态、可行性、锁定、槽位是否混为一个字段。
- 车辆是否误入 V2 匹配逻辑。
- UI 是否隐藏不可行预警或只靠颜色表达。

### P2 建议

- 组件是否遵守既有 token 和信息密度。
- 是否增加未被 PRD 要求的抽象或配置表。
- 是否有可删除的 V1 兼容分支；删除前必须有迁移验收和用户批准。
