## 1. 总体原则

### 1.1 核心原则

1. **永远不阻断业务流程** — 高德 API 失败时，系统必须降级运行，不允许异常中断页面或流程。
2. **能缓存就缓存** — 地理编码结果永久缓存（地址不变则坐标不变），ETA 结果短期缓存（司机位置随时变化）。
3. **先本地后远程** — 路径规划前先通过本地预筛减少候选司机数，只对 Top K 候选调用高德。
4. **成本可控** — 每次 API 调用都是成本（配额消耗），必须能用数字说明调用量。

### 1.2 API 调用全景

| 场景 | API | 调用时机 | 频率特征 | 缓存策略 |
|------|-----|---------|---------|---------|
| 地理编码 | `/v3/geocode/geo` | 订单导入、地址变更 | 低频（每订单 2 次） | RDS 永久缓存 |
| 驾车路径规划 | `/v3/direction/driving` | 派单推荐、调度确认 | 中频（每候选 1 次） | Tair 短期缓存（30-120s） |
| 地图渲染 | JS API 2.0 | 页面加载 | 高频（每 PV 1 次） | 浏览器 CDN 缓存 |

## 2. 地理编码策略

### 2.1 调用时机

**必须调用：**

- 订单导入时，对取车地址和还车地址分别调用（当前实现见 `src/lib/import/services/geocode.ts`）。
- 地址变更时（运营修改订单地址），重新调用并更新缓存。

**禁止调用：**

- 订单已有有效坐标（`pickupLat/pickupLng` 和 `returnLat/returnLng` 均非空且地址文本未变）。
- 地址已命中 `geocode_cache` 表（见 2.3）。

### 2.2 调用流程

```
地址输入
  │
  ├─ 归一化地址文本（去空格、统一全角半角、去冗余后缀）
  │
  ├─ 查 geocode_cache 表
  │     │
  │     ├─ 命中 → 直接使用缓存坐标，不调用高德
  │     │
  │     └─ 未命中 → 调用高德 /v3/geocode/geo
  │           │
  │           ├─ 成功 → 写入 geocode_cache，返回坐标
  │           │
  │           └─ 失败 → 订单入库，标记"需复核"，进入预警中心
  │
  └─ 订单入库（无论地理编码是否成功）
```

### 2.3 缓存设计：`geocode_cache` 表

在 PostgreSQL 中新建缓存表，**永久复用**（地址不变则坐标不变）。

```sql
CREATE TABLE geocode_cache (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    address     TEXT NOT NULL,         -- 归一化后的地址文本
    lat         DOUBLE PRECISION NOT NULL,
    lng         DOUBLE PRECISION NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX idx_geocode_cache_address ON geocode_cache (address);
```

缓存读取逻辑：

```ts
// 伪代码：地理编码前先查缓存
async function geocodeWithCache(address: string, label: string) {
  const normalized = normalizeAddress(address);
  const cached = await prisma.geocodeCache.findUnique({
    where: { address: normalized }
  });
  if (cached) return { success: true, lat: cached.lat, lng: cached.lng };

  const result = await geocodeAddress(normalized, label);
  if (result.success) {
    await prisma.geocodeCache.upsert({
      where: { address: normalized },
      create: { address: normalized, lat: result.lat, lng: result.lng },
      update: { lat: result.lat, lng: result.lng, updatedAt: new Date() }
    });
  }
  return result;
}
```

### 2.4 地址归一化规则

归一化在写入缓存前执行，保证同一地址的不同写法命中同一缓存：

| 操作 | 示例 |
|------|------|
| 去首尾空格 | `" 上海市浦东新区 "` → `"上海市浦东新区"` |
| 全角数字转半角 | `"１２３号"` → `"123号"` |
| 全角字母转半角 | `"Ａ座"` → `"A座"` |
| 去尾部冗余词 | `"xxx（近地铁站）"` → `"xxx"` |
| 统一标点 | 中文括号 `（）` 不转换，保留原始语义 |

> 归一化函数实现在 `src/lib/import/services/address-normalize.ts`（新建）。

### 2.5 失败处理

| 失败类型 | 处理方式 | 用户可见 |
|---------|---------|---------|
| `AMAP_KEY_MISSING` | 订单入库，坐标置 NULL | 预警中心提示"未配置高德 Key" |
| `GEOCODE_FAILED` / 超时 | 订单入库，坐标置 NULL | 预警中心提示"地址 X 地理编码失败，需复核" |
| 网络错误 | 最多重试 1 次（间隔 1s），仍失败同上 | 同 GEOCODE_FAILED |

**关键设计：** 地理编码失败不阻断订单导入。订单正常入库，`status=PENDING`，但缺少坐标信息，需运营人员在预警中心手动补全。

### 2.6 当前实现差距

当前 `geocode.ts` 已实现基础的 API 调用和超时控制（4s），缺少以下能力（计划在后续迭代补齐）：

- [ ] 地址归一化
- [ ] `geocode_cache` 表查询与写入
- [ ] 重试逻辑
- [ ] 预警中心集成（failed_geocode 事件写入操作日志或独立预警表）
- [ ] traceId 贯穿（日志中携带 traceId）

## 3. 路径规划与 ETA 策略

### 3.1 整体流程

```
调度请求
  │
  ├─ 1. 本地预筛（filter.ts）
  │     ├─ 状态过滤：仅 S1/S2/S3/S4
  │     ├─ 同门店过滤：仅同门店司机
  │     ├─ 负载过滤：上门订单限制活跃上门单数
  │     └─ 输出候选列表（长度 M）
  │
  ├─ 2. 候选排序（sort.ts）
  │     └─ 按直线距离 + 负载惩罚排序，取 Top K（K 默认 5）
  │
  ├─ 3. 缓存检查
  │     └─ 查 Tair：eta:{orderId}:{driverId}:driving
  │
  ├─ 4. 调用高德驾车路径规划（仅对 Top K 中缓存未命中的候选）
  │     ├─ 成功 → 写入 Tair（TTL 30-120s）
  │     └─ 失败 → ETA = 9999 分钟，标记"需人工判断"
  │
  └─ 5. 输出排序结果 + 推荐理由
```

### 3.2 何时调用

| 场景 | 是否调用 | 说明 |
|------|---------|------|
| 调度员点击"推荐司机" | 是 | 对 Top K 候选逐一调用 |
| 调度确认（confirm） | **强制刷新** | 无视缓存，确保 ETA 是最新值（防止司机位置在缓存期间大幅偏移） |
| 地图看板自动刷新 | 否 | 仅展示已有数据，不触发路径规划 |
| 订单列表页 | 否 | 不触发任何高德 API |

### 3.3 缓存设计：Tair（Redis）

**Key 格式：** `eta:{orderId}:{driverId}:driving`

**Value 结构（JSON）：**

```json
{
  "etaMinutes": 15,
  "durationSeconds": 872,
  "distanceMeters": 5200,
  "fetchedAt": "2026-07-04T10:30:00Z"
}
```

**TTL（过期时间）：**

| 场景 | TTL | 理由 |
|------|-----|------|
| 调度推荐 | 120 秒 | 允许短时间内多次推荐复用结果 |
| 调度确认后 | 30 秒 | 确认后立即缩短，防止旧数据被后续查询命中 |
| 降级场景（API 不可用） | 300 秒 | 延长缓存生命周期，减少无效重试 |

**缓存操作流程：**

```
读取：查 Tair → 命中且未过期 → 返回缓存值（跳过 API 调用）
写入：API 调用成功 → SET key value EX ttl
强制刷新（confirm 场景）：DEL key → 重新调用 API → SET key value EX ttl
```

### 3.4 Top K 策略

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `topNLimit` | `3`（引擎默认值） | 排序后返回给调度员的候选数 |
| `topKForEta` | `5` | 调用高德 ETA 的最大候选数。K >= N，因为部分候选可能因 ETA 过长被降级 |

**选择逻辑：** 先对全部候选按直线距离 + 负载惩罚排序，取前 K 名调用高德，返回 Top N 给调度员。

### 3.5 ETA 阈值

| 阈值 | 含义 | 处理 |
|------|------|------|
| ETA < 120 分钟 | 可自动派单 | 正常返回排序列表 |
| ETA >= 120 分钟 | 超过可接受范围 | 标记 `MANUAL`，推荐理由标注"距离过远，建议人工判断" |
| ETA = 9999 | API 调用失败 | 标记为降级候选，推荐理由标注"ETA 获取失败，需人工判断" |

### 3.6 fallback 估算

当高德 API 不可用时，使用基于司机状态的静态估算（当前实现见 `eta.ts`）：

```
OFFLINE      → 9999 分钟
UNAVAILABLE  → 9999 分钟
S1           → 18 + jitter 分钟
S2           → 28 + jitter 分钟
S3           → 42 + jitter 分钟
S4           → 58 + jitter 分钟
```

`jitter` 由 `driverId` 字符码稳定计算，保证同一司机每次估算结果一致。

### 3.7 当前实现差距

当前 `eta.ts` 已实现基础的 API 调用、超时控制（5s）、失败降级（9999）和 fallback 估算。缺少以下能力：

- [ ] Tair 缓存读写
- [ ] 调度确认前的强制刷新
- [ ] Top K 截断（当前对所有候选逐一调用）
- [ ] 请求队列和并发控制
- [ ] traceId 在日志中的完整贯穿

## 4. 限流与配额

### 4.1 高德免费额度（截至 2026 年 7 月）

| API | 个人开发者日限额 | 企业开发者日限额 | QPS 上限 |
|-----|----------------|----------------|---------|
| 地理编码 (`/geocode/geo`) | 5,000 | 300,000 | 30 |
| 驾车路径规划 (`/direction/driving`) | 5,000 | 300,000 | 30 |
| JS API 2.0（地图渲染） | 免费（非商用） | 免费（非商用） | N/A（按 PV 计） |

> 数据来源：高德开放平台官方文档。企业认证后可大幅提升额度。建议 MVP 验证通过后完成企业认证。

### 4.2 QPS 限制与服务端保护

高德 Web 服务 API 默认 QPS 上限约 30。为防止突发流量触发限流，在应用层增加如下保护：

#### 4.2.1 请求队列

单个 Node.js 进程内维护一个请求队列，限制并发调用数：

```ts
// 伪代码：请求队列示意
const amapQueue = new PQueue({
  concurrency: 5,        // 最多 5 个并发请求
  interval: 1000,        // 每秒
  intervalCap: 20        // 每秒最多 20 个请求（留 10 QPS 余量）
});
```

#### 4.2.2 并发控制

| 场景 | 并发策略 | 原因 |
|------|---------|------|
| 订单导入（地理编码） | 串行处理地址（一个订单的两个地址顺序调用） | 低频，无需并发 |
| 批量导入 | 每批最多 10 个地址并发 | 控制突发 QPS |
| ETA 查询（Top K） | 最多 5 个并发（与 Top K 对齐） | 候选数量本身就是限制 |

### 4.3 重试策略（指数退避）

```
第 1 次失败 → 等待 1s 后重试
第 2 次失败 → 等待 2s 后重试
第 3 次失败 → 放弃，走降级逻辑
```

- 总重试次数：最多 3 次（含首次调用共 4 次尝试）
- 仅对网络错误和 5xx 重试；高德返回的业务错误（如 `status=0`）不重试
- 超时错误（`AbortError`）视为网络错误，参与重试
- 每次重试前检查是否已超过全局超时（见 4.4）

### 4.4 全局超时

| API | 当前超时 | 含重试总超时 | 说明 |
|-----|---------|------------|------|
| 地理编码 | 4s | 8s | 含 1 次重试 |
| 驾车路径规划 | 5s | 12s | 含最多 3 次重试 |

### 4.5 超出配额时的降级方案

当日调用量接近或超出免费配额时：

| 阶段 | 触发条件 | 降级措施 |
|------|---------|---------|
| 预警 | 日调用量达到配额的 80% | 日志告警 + 控制台提示 |
| 软限制 | 日调用量达到配额的 95% | 停止非必要调用（地图看板自动刷新），仅保留调度确认和订单导入 |
| 硬限制 | API 返回配额耗尽错误（`infocode=10006`） | 全部走降级逻辑（ETA=9999，地理编码走缓存兜底） |

## 5. 降级策略

### 5.1 总体策略：不阻断、不抛异常、可观测

所有高德 API 调用失败时：
- **不抛未捕获异常** — 所有错误在模块内部消化
- **不阻断业务流程** — 页面正常渲染，操作正常执行
- **可观测** — 日志记录降级原因，预警中心展示待处理项

### 5.2 各场景降级行为

#### 5.2.1 路径规划失败

```
API 返回非 200 / 超时 / status != "1"
  → ETA = 9999 分钟
  → 日志记录：dispatch_eta_degraded + 失败原因
  → 推荐理由标注："ETA 获取失败，需人工判断"
  → 调度员可手动派单（忽略 ETA）
```

#### 5.2.2 地理编码失败

```
API 返回非 200 / 超时 / status != "1"
  → 订单入库（status = PENDING）
  → pickupLat/pickupLng 或 returnLat/returnLng 置 NULL
  → 日志记录：import_geocode_failed + 地址文本
  → 预警中心展示："取车地址/还车地址地理编码失败，需人工补全坐标"
  → 订单可在预警中心手动编辑补全坐标后进入正常流转
```

#### 5.2.3 JS API 地图加载失败

```
JS API 脚本加载失败 / _AMapSecurityConfig 未配置
  → 回退到本地降级视图（网格背景 + 文字提示）
  → 保留当前已有的降级地图组件
  → 不影响订单列表、派单等其他功能
```

### 5.3 降级状态码速查

| 码 | 含义 | 来源 |
|----|------|------|
| `ETA_FAILURE_MINUTES = 9999` | 路径规划失败 | `src/lib/dispatch/eta.ts` |
| `AMAP_KEY_MISSING` | 服务端 Key 未配置 | `src/lib/import/services/geocode.ts` |
| `GEOCODE_FAILED` | 地理编码失败 | `src/lib/import/services/geocode.ts` |
| `ORIGIN_MISSING` | 司机无 GPS 位置 | `src/lib/dispatch/eta.ts` |

### 5.4 日志规范

所有降级事件必须记录结构化日志，字段至少包含：

```ts
{
  event: "dispatch_eta_degraded" | "import_geocode_failed",
  traceId: string | null,     // 全链路追踪 ID
  driverId?: string,           // ETA 场景
  address?: string,            // 地理编码场景
  reason: string,              // 失败原因（AMAP_KEY_MISSING / AMAP_HTTP_500 / AMAP_PAYLOAD_INVALID 等）
  timestamp: string            // ISO 8601
}
```

## 6. Key 管理

### 6.1 Key 用途与隔离

| 环境变量 | API 类型 | 用途 | 调用位置 | 平台选择 |
|---------|---------|------|---------|---------|
| `AMAP_SERVER_KEY` | Web 服务 API | 地理编码、驾车路径规划 | 服务端 `fetch()` | **"Web服务"** |
| `NEXT_PUBLIC_AMAP_JS_KEY` | Web 端 JS API | 浏览器地图渲染 | 前端 `<script>` 加载 | **"Web端(JS API)"** |
| `NEXT_PUBLIC_AMAP_SECURITY_JS_CODE` | JS API 安全密钥 | 请求签名（2021.12.02 后的 Key 必须） | `window._AMapSecurityConfig` | 与 JS Key 配套 |

> **两个 Key 不能共用。** 详细申请步骤见 `docs/amap-key-guide.md`。

### 6.2 安全存储

| 环境 | 存储方式 |
|------|---------|
| 本地开发 | `.env.local`（已加入 `.gitignore`，不提交） |
| CI/CD | GitHub Secrets / Vercel Environment Variables |
| 生产环境（Railway） | Railway Variables（加密存储） |

**安全建议：**

- 服务端 Key（`AMAP_SERVER_KEY`）永远不在前端代码中引用，不使用 `NEXT_PUBLIC_` 前缀。
- 生产环境 JS API Key 在高德控制台配置**域名白名单**，限制仅允许生产域名调用。
- 定期在高德控制台查看 Key 用量，发现异常调用立即轮换。

### 6.3 Key 轮换流程

1. 在高德控制台创建新 Key（与原 Key 同应用、同平台类型）。
2. 更新环境变量，保留旧 Key 作为备份。
3. 观察新 Key 调用成功率（至少 1 小时）。
4. 确认无异常后，在高德控制台删除旧 Key。
5. 更新文档记录轮换时间。

## 7. 成本估算

### 7.1 假设条件

| 参数 | 保守估计 | 乐观估计 |
|------|---------|---------|
| 日订单量 | 50 | 500 |
| 单均地址数（取车 + 还车） | 2 | 2 |
| 缓存命中率（地理编码） | 60%（重复地址/区域） | 85% |
| 需调度订单比例 | 80% | 70% |
| 每次调度候选数（Top K） | 5 | 5 |
| 改派率 | 10%（每单平均改派次数） | 5% |
| 调度确认强制刷新 | 每次确认 +1 次调用/候选 | 同左 |

### 7.2 日调用量估算

#### 地理编码

| 估算项 | 保守 | 乐观 |
|--------|------|------|
| 导入调用（未命中缓存） | 50 × 2 × 40% = 40 | 500 × 2 × 15% = 150 |
| 地址变更调用 | 5 | 20 |
| **日合计** | **45** | **170** |

#### 路径规划（ETA）

| 估算项 | 保守 | 乐观 |
|--------|------|------|
| 调度推荐 | 50 × 80% × 5 = 200 | 500 × 70% × 5 = 1,750 |
| 改派重算 | 200 × 10% = 20 | 1,750 × 5% = 88 |
| 确认强制刷新 | 50 × 80% × 5 = 200 | 500 × 70% × 5 = 1,750 |
| **日合计** | **420** | **3,588** |

#### 总计

| 估算项 | 保守 | 乐观 |
|--------|------|------|
| 日 API 调用总量 | **465** | **3,758** |

### 7.4 缓存带来的成本节省

## 9. 附录：API 端点速查

### 9.1 地理编码

```
GET https://restapi.amap.com/v3/geocode/geo
  ?key={AMAP_SERVER_KEY}
  &address={地址文本}

响应：
{
  "status": "1",        // 1=成功, 0=失败
  "infocode": "10000",  // 10000=正常
  "geocodes": [{
    "location": "121.473701,31.230416"  // lng,lat
  }]
}
```

### 9.2 驾车路径规划

```
GET https://restapi.amap.com/v3/direction/driving
  ?key={AMAP_SERVER_KEY}
  &origin={lng},{lat}
  &destination={lng},{lat}

响应：
{
  "status": "1",
  "route": {
    "paths": [{
      "duration": "872",    // 预计耗时（秒）
      "distance": "5200"    // 距离（米）
    }]
  }
}
```

### 9.3 错误码速查

| infocode | 含义 | 处理 |
|----------|------|------|
| `10000` | 正常 | - |
| `10001` | Key 不存在或已过期 | 检查 `AMAP_SERVER_KEY` |
| `10003` | Key 未开通对应平台服务 | 检查 Key 的"服务平台"类型 |
| `10006` | 配额耗尽 | 走降级逻辑 |
| `20000` | 请求参数非法 | 检查传入的坐标/地址格式 |
| `30000` | 服务端内部错误 | 走重试逻辑 |
